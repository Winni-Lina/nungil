# 눈길(Nungil) 기술 보고서

**프로젝트명**: 눈길 — 지적장애인 생활보조 AI 어시스턴트  
**개발 기간**: 2026년 3월 ~ 7월  
**팀 구성**: 2인 (클라이언트 / 서버 협업)  
**리포지토리**: https://github.com/Winni-Lina/nungil

---

## 1. 프로젝트 개요

### 1.1 배경

지적장애인은 손씻기, 빨래, 외출 준비처럼 일반인에게 익숙한 일상 과업도 단계별로 막히면 중도 포기하는 경우가 많다. 기존 보조 앱들은 미리 정해진 동영상이나 문자 안내를 보여주는 수준이었고, 사용자별 인지 수준·장소·특이사항을 반영한 맞춤 안내는 제공되지 않았다. 보호자는 하루에도 수차례 "이거 어떻게 해?"라는 전화를 받아야 했다.

눈길은 이 문제를 생성형 AI로 해결하려는 시도다. 보호자가 일정을 등록하면 AI가 맞춤 단계를 자동으로 만들고, 사용자는 "똘똘아"라는 웨이크워드 하나로 AI 어시스턴트를 불러 음성 대화만으로 과업을 진행한다.

### 1.2 목표

- 지적장애인이 보호자 없이 일상 과업을 스스로 완수하도록 지원
- 보호자의 반복 개입 부담을 최소화
- AI의 역할을 '안내와 대응'으로 한정해 환각(hallucination)의 영향을 통제

### 1.3 팀 구성 및 역할

| 구성원 | 주 담당 |
|---|---|
| 동호 | 서버(Spring, MyBatis, Gemini 연동), AI 아키텍처 설계 |
| 리나 | 클라이언트(Android Kotlin), UI/UX, 알람·FCM |

## 2. 서비스 개요

### 2.1 주요 사용자

- **사용자(지적장애인)**: 앱 음성 안내를 따라 과업 수행. 글보다 음성 위주.
- **보호자**: 일정 등록, AI 생성 단계 검토, 사용자 진행 상황 모니터링.

### 2.2 핵심 기능

| 기능 | 설명 |
|---|---|
| 보호자 회원가입 / 로그인 | 아이디 중복 확인, 임시 비밀번호 발급 |
| QR 1:1 페어링 | 보호자가 QR 발급 → 사용자 앱에서 스캔하면 연동 완료 |
| 일정 등록 + AI 단계 자동 생성 | 과업·장소·특이사항 입력 → Gemini가 맞춤 단계 생성 → 보호자 검토·편집·확정 |
| "똘똘아" 웨이크워드 | Vosk 온디바이스 STT, 인터넷 없이 동작 |
| 단계별 음성 안내 | Google Cloud STT로 음성 입력 → Gemini 응답 → TTS 출력 |
| 완료 키워드 감지 | "했어 / 응 / 됐어" 등 정확 매칭 → 자동 다음 단계 |
| 카메라 멀티모달 질문 | "이만큼이면 돼?" → 사진 촬영 → Gemini 시각 분석 → 음성 답변 |
| FCM 실시간 알림 | 미수행·반복 질문 발생 시 보호자에게 푸시 |
| 수행 리포트 | 일별 완료율, 14일 과업 추세 차트 |
| 오늘 일정 탭 | 사용자가 채팅 화면에서 버튼 하나로 오늘 일정 전체 확인 |

### 2.3 서비스 흐름 요약

```
보호자: 일정 등록 → AI 단계 생성 → 검토 저장
                                       ↓ FCM 알림
사용자: "똘똘아" → 과업 시작 → 단계별 음성 안내 → 완료
                                       ↓
보호자: 수행 리포트 확인
```

## 3. 설계

### 3.1 시스템 아키텍처

하나의 Android APK 안에 보호자 앱과 사용자 앱이 공존한다. 역할 선택은 첫 실행 시 결정하며, QR 연동 이후 분리된다.

```
[보호자 앱] ──── REST API ──── [Spring MVC 서버 / Tomcat]
[사용자 앱] ──── REST API ─────┤
                                ├── Oracle DB
                                ├── Google Gemini 2.5 Flash (REST)
                                ├── Google Cloud STT
                                └── Firebase FCM ──→ [사용자 앱]
```

### 3.2 AI 설계 — "AI를 통제하는 구조"

눈길의 핵심 설계 원칙은 AI에게 모든 것을 맡기지 않는 것이다. 생성형 AI는 강력하지만 비결정적이다. 지적장애 사용자에게는 "예측 가능하고 일관된" 것이 "똑똑하지만 가끔 틀리는" 것보다 훨씬 중요하다.

**AI 호출은 세 시점으로만 제한한다:**

| 호출 시점 | 역할 |
|---|---|
| 보호자 일정 등록 시 (1회) | 과업 + 장소 + 특이사항 → 맞춤 단계 생성 |
| 자유 채팅 중 | 질문 답변 + 사진 필요 여부 판단 |
| 일정 수행 중 매 발화 | 현재 단계 안내 + 4분류 대응 (완료신호 / 질문 / 딴소리 / 힘들어함) |

**완료 판단은 AI가 아니라 앱이 한다.** 앱이 먼저 완료 키워드(정확 일치 5개 + 포함 9개)를 확인하고, 그 외만 서버로 넘긴다.

**변경 전 vs 변경 후:**

| 항목 | 변경 전 | 변경 후 |
|---|---|---|
| 단계 출처 | 과업 공통 고정 단계 (개인화 없음) | 등록 시 AI가 장소·특이사항 반영해 맞춤 생성, DB 저장 |
| 완료 판단 | AI가 판단 (오판 잦음) | 앱이 키워드 직접 감지 (결정적) |
| 프롬프트 구조 | 단일 프롬프트 (규칙이 입력에 섞임) | system / user 분리 (규칙 준수·일관성 향상) |

### 3.3 프롬프트 엔지니어링 기법

| 기법 | 효과 |
|---|---|
| `system_instruction` / `user` 역할 분리 | 규칙 준수·일관성·프롬프트 인젝션 저항 |
| `responseSchema` 구조화 출력 | JSON 4필드 강제 반환 → 파싱 안정성 |
| `responseMimeType=application/json` | 마크다운 펜스 없이 순수 JSON 반환 |
| `thinkingBudget=0` | 내부 추론 비활성 → 지연·비용 절감 |
| `maxOutputTokens=512` | 짧은 답변 강제 (사용자 특성 고려) |
| 4분류 행동 지침 명시 | AI 응답을 안내·대응 범위로 한정 |

### 3.4 알람 다층 안전망

지적장애 사용자에게 알람 누락은 곧 일정 실패다. 세 단계로 방어한다.

- **1차**: `AlarmManager.setExactAndAllowWhileIdle` — Doze 모드를 관통하는 정시 발동
- **2차**: WorkManager 15분 주기 — 5~60분 지연 구간 "늦은 알림" 백업
- **3차**: 재부팅 복구 — BootReceiver가 즉시 동기화 + 주기 워커 재등록

### 3.5 데이터베이스 설계 (ERD 주요 테이블)

| 테이블 | 역할 |
|---|---|
| `GUARDIAN` | 보호자 계정 |
| `NUNGIL_USER` | 사용자(지적장애인) + 보호자 FK |
| `SCHEDULE` | 일정 (과업, 시간, 상태, custom_steps JSON) |
| `QUESTION_LOG` | 대화 로그 (사용자 발화 + AI 응답) |
| `TASK` | 과업 마스터 (빨래, 손씻기 등) |

| 모드 | 입력 | 기대 동작 | 결과 |
|---|---|---|---|
| 채팅 | "이게 뭐야?" | photoRequest=true | ✅ 통과 |

## 6. 회고

### 6.1 프로젝트 전체 회고

이 프로젝트를 통해 가장 크게 깨달은 것은 **"AI를 쓴다"는 것과 "AI가 실제로 잘 작동한다"는 것은 다르다**는 점이다.

처음 설계는 단순했다. AI가 단계를 만들고 안내하고 완료도 판단한다. 개발을 시작하자마자 문제가 쌓였다. 사용자가 "이거 다 했는데 맞아?"라고 물으면 AI는 완료 신호를 보냈다. 같은 과업인데도 AI가 생성하는 단계가 매번 달라서 사용자가 혼란스러워했다. 빠른 응답이 필요한 순간에 Gemini 응답을 기다리느라 지연이 생겼다.

결국 구조를 두 번 바꿨다. "생성 시점"과 "실행 시점"을 분리하고, 완료 판단을 앱의 키워드 감지로 이관했다. 이 결정 이후 안정성이 눈에 띄게 올라갔다. 모델 자체보다 **프롬프트 구조와 역할 분리, 응답 범위 제한**이 결과물의 품질을 결정한다는 것을 직접 경험했다.

알람 문제도 예상보다 복잡했다. Android 12 이상에서 정확한 알람 권한이 별도로 필요하다는 것을 뒤늦게 알았고, 삼성 기기와 다른 제조사에서 배터리 최적화 동작이 달라 같은 코드가 다르게 실행됐다. AlarmManager + WorkManager + BootReceiver 3중 구조로 방어했지만, 결국 "알람이 안 울리면 눌러요" 버튼을 추가하는 현실적인 선택도 해야 했다. 완벽한 기술적 해결이 불가능할 때 사용자 경험으로 보완하는 방법을 고민하게 됐다.

접근성 제약이 설계를 오히려 더 명확하게 만들었다. 화면은 크게, 글씨는 굵게, 응답은 짧고 구체적으로 — 이 원칙들이 일반 사용자에게도 좋은 UX로 이어진다는 것을 확인했다. 타깃 사용자를 뚜렷하게 정하는 것이 설계를 단순하게 해준다.

### 6.2 개인 회고

**동호 (서버 / AI 담당)**

서버를 처음 짰을 때는 AI 호출이 여기저기 흩어져 있었다. 일정 조회할 때도 AI, 단계 진행할 때도 AI, 완료 판단도 AI에게 물었다. 나중에 보니 관심사 분리가 전혀 안 된 구조였다. AnalysisOrchestrator를 만들어 AI 진입점을 하나로 모으고, StepGenerationService를 분리해 단계 생성 로직을 독립시켰다. 코드 정리를 하면서 테스트를 먼저 쓰고 나서야 "이 함수가 뭘 하는지"가 명확해졌다. 테스트가 설계를 개선하는 도구라는 것을 이번에 처음 실감했다.

Vertex AI 전환도 처음엔 겁났다. AI Studio 키에서 Vertex AI 서비스 계정 인증으로 바꾸는 과정에서 인증 오류가 반복됐고, Google Cloud 문서를 여러 번 읽어야 했다. 결국 STT와 같은 서비스 계정을 재사용하는 방법으로 해결했다. 문제를 우회하는 방법보다 구조를 이해하고 제대로 해결하는 방법이 훨씬 깔끔하다는 것을 다시 배웠다.

**리나 (클라이언트 / UI 담당)**

Android 알람이 이렇게 복잡한 줄 몰랐다. `setExact`를 쓰면 되는 줄 알았는데 Doze 모드, 배터리 최적화, 기기별 동작 차이 때문에 알람이 안 울리는 경우가 계속 생겼다. WorkManager를 추가하고 BootReceiver를 달고 나서야 어느 정도 안정됐다. 완벽한 해결이 없다는 것을 받아들이는 것도 개발의 일부라는 걸 알았다.

UI를 만들면서 "사용자가 글을 못 읽으면 어떡하지?"를 계속 생각했다. 버튼에 글자 대신 아이콘을 추가하고, 응답을 최대한 짧게 쓰고, 완료 버튼을 크게 만들었다. 교수님 피드백에서 "왜 이렇게 설계했나?"라는 질문을 받을 때 대답할 수 있으려면 근거가 있어야 했고, 그래서 선행 연구도 찾아보게 됐다. 기능을 만드는 것과 왜 만드는지 설명할 수 있는 것은 다른 능력이라는 걸 배웠다.
