# 눈길(Nungil) 프로젝트 기술 보고서

**프로젝트명**: 눈길 — 지적장애인 생활보조 AI 어시스턴트  
**개발 기간**: 2026년 3월 ~ 7월  
**팀 구성**: 2인 (클라이언트/서버 협업)  
**리포지토리**: https://github.com/Winni-Lina/nungil

---

## 1. 프로젝트 개요

눈길은 중등도 지적장애인이 보호자 없이도 일상 과업(양치, 빨래, 외출 준비 등)을 수행할 수 있도록 AI가 음성으로 단계별 안내를 제공하는 듀얼 앱이다. 보호자는 일정을 사전에 등록하고 AI가 자동 생성한 단계를 검토하며, 사용자(장애인)는 "똘똘아"라는 웨이크워드 한 마디로 AI 어시스턴트를 호출해 대화만으로 과업을 진행한다.

프로젝트를 시작할 때 가장 먼저 부딪힌 질문은 "AI를 어느 시점에, 어떻게 개입시킬 것인가"였다. 단순히 챗봇을 붙이는 것으로는 장애 사용자의 실제 요구를 충족하기 어렵다고 판단했고, 그 고민이 이후 개발 방향 전체를 결정지었다.

## 2. 기술 스택 선택 이유

| 구성 요소 | 선택 | 이유 |
|---|---|---|
| AI 모델 | Google Gemini 2.5 Flash | JSON 스키마 강제 응답(responseSchema) 지원 — 구조화된 판단 결과를 안정적으로 받기 위함 |
| STT | Google Cloud STT + Vosk(오프라인) | Google STT는 정확도, Vosk는 웨이크워드 감지에 인터넷 불필요 |
| 백엔드 | Spring MVC + MyBatis + Oracle | 팀 역량과 기존 인프라 고려 |
| 클라이언트 | Android Kotlin | 타깃 사용자 환경(안드로이드 보급률) |
| 푸시 알림 | Firebase FCM | 일정 시간에 사용자에게 알림을 보내는 가장 신뢰성 높은 방법 |

Gemini 선택은 단순한 문장 생성 이상의 이유가 있었다. `stepComplete`, `photoRequest`, `suggestedQuestions` 같은 필드를 JSON으로 강제 반환받아야 앱이 다음 동작(단계 진행, 카메라 호출 등)을 결정할 수 있기 때문이다. 이 구조화 응답 없이는 AI 답변을 파싱하는 로직이 너무 취약해진다.

## 3. 개발 과정에서 겪은 주요 어려움

### 3.1 AI 구조를 두 번 바꿨다

가장 많은 시간을 쓴 문제였다.

**처음 설계**는 사용자가 일정을 시작하면 그 순간 AI가 단계를 생성하고, 동시에 안내까지 담당하는 구조였다. 개발하면서 바로 문제가 드러났다. AI가 단계를 직접 생성하기 때문에 보호자가 내용을 사전에 확인할 수 없었고, 매 단계 전환마다 Gemini를 다시 호출하니 응답 지연이 쌓였다. 무엇보다, 단계 수나 내용이 실행할 때마다 달라지는 경우가 생겨 사용자가 혼란스러워했다.

**두 번째 구조**는 "생성 시점"과 "실행 시점"을 분리한 것이다. 보호자가 일정을 등록할 때 AI가 맞춤 단계를 한 번 생성하고, 보호자가 이를 검토·확정한다. 실행 시점에 AI는 단계 안내와 사용자 발화 해석만 담당한다. 이 구조로 바꾸고 나서 단계 일관성 문제가 해소됐고, AI 호출 횟수도 줄었다.

### 3.2 `stepComplete` 오판 버그

사용자가 "이거 다 했는데 맞아?" 라고 물으면 AI가 완료 신호(`stepComplete: true`)를 보내는 버그가 반복됐다. 질문형 발화를 완료 발화로 오인하는 것이었다.

프롬프트에 "의문문이 포함된 경우 반드시 `stepComplete: false`를 반환하라"는 규칙을 명시했지만 한 번에 해결되지 않았다. 결국 두 가지 방법을 함께 적용했다. 첫째, system instruction에 역할과 규칙을 두고 user prompt에 실제 대화 데이터를 분리했다(교수님 피드백 반영). 둘째, "질문형 발화는 사진 요청으로 유도하라"는 fallback 동작을 정의했다. 이후 실제 서버에서 8가지 시나리오를 테스트한 결과 7/8 통과를 확인했으며, "이거 다 했는데 맞아?" 케이스는 정상 처리되는 것을 검증했다.

### 3.3 알람이 울리지 않는 문제

Android 12 이상에서 정확한 알람(`setExactAndAllowWhileIdle`) 권한이 필요하다는 사실을 뒤늦게 발견했다. 권한 없이 설정된 알람은 시스템에 의해 지연되거나 무시됐다. 기기마다 배터리 최적화 예외 설정이 달라서 삼성 기기에서만 안 울리는 상황도 있었다.

해결책은 세 단계였다. 권한 체크 후 `canScheduleExactAlarms()`로 분기, 권한이 없을 때는 `setAndAllowWhileIdle`로 fallback, WorkManager 15분 주기 체크로 놓친 알람 보완. 그럼에도 완벽하지 않아서 사용자 설정 화면에 "알람이 안 울리면 눌러요" 동기화 버튼을 추가했다. 알람은 100% 보장이 어렵다는 Android 플랫폼의 현실적 한계를 인정한 설계다.

### 3.4 보호자-사용자 연결 구조

처음에는 보호자가 사용자 ID를 직접 입력하는 방식을 생각했다. ID를 몰라서 연결이 안 되는 경우가 테스트 중 여러 번 생겼다. QR 코드로 방향을 바꿨다. 보호자 앱에서 QR를 생성하면 사용자가 스캔하는 방식으로, 연결 과정을 1분 이내로 줄였다.

### 3.5 웨이크워드 인식과 배터리 소모

Vosk 오프라인 STT를 웨이크워드 감지용으로 채택했는데, 마이크를 상시 열어두는 방식이라 배터리 소모가 적지 않았다. 앱이 백그라운드에 있을 때 서비스가 종료되는 문제도 있었다. Foreground Service로 전환하고 알림을 유지하는 방식으로 해결했다. 여전히 배터리 소모가 있어 향후 개선이 필요한 부분으로 남아 있다.

## 4. AI 프롬프트 성능 평가 결과

실제 운영 서버에 8가지 시나리오를 직접 호출해 검증했다.

| 모드 | 테스트 | 통과 |
|---|---|---|
| 자유채팅 | 3건 | 3/3 |
| 일정수행 | 5건 | 4/5 |
| **합계** | **8건** | **7/8 (87.5%)** |

유일하게 부분통과한 케이스는 "이거 너무 힘들어 못하겠어"에 대한 응답이었다. 시스템 프롬프트는 이 경우 단계를 더 잘게 쪼개서 재안내하도록 지시하고 있으나, 실제 응답은 격려 멘트만 추가하고 단계를 세분화하지는 않았다. 프롬프트 보강이 필요한 부분으로, 후속 과제로 남겨두었다.

## 6. 한계 및 향후 개선 과제

1. **배터리 최적화**: 웨이크워드 상시 감지로 인한 배터리 소모는 현재 해결하지 못한 문제다. Edge TPU 등 경량 모델로 전환을 검토할 수 있다.
2. **AI 힘듦 대응**: "힘들어" 발화 시 단계 세분화 응답이 아직 불안정하다. 프롬프트 추가 보강 필요.
3. **오프라인 동작**: 현재 AI 기능은 인터넷 연결이 필수다. 온디바이스 LLM 도입 시 오프라인 사용이 가능해진다.
4. **단계 개인화**: 현재 AI가 생성한 단계를 보호자가 수동으로 수정할 수 있지만, 사용자의 과거 수행 이력을 기반으로 단계를 자동 조정하는 기능이 있으면 더 효과적일 것이다.

## 7. 결론

이 프로젝트를 통해 "AI를 쓴다"는 것과 "AI가 실제로 잘 작동한다"는 것 사이의 간극을 직접 경험했다. 모델 자체보다 프롬프트 설계, 응답 구조화, 엣지 케이스 처리가 결과물의 품질을 결정한다는 점을 배웠다. 알람 신뢰성 문제처럼 플랫폼의 한계를 우회하는 설계도 중요하다는 것을 알았다. 

지적장애인 사용자를 위한 앱이라는 제약이 오히려 설계를 더 단순하고 명확하게 만들어줬다. 화면은 크고 글씨는 굵게, 응답은 짧고 구체적으로 — 접근성 설계 원칙이 일반 사용자에게도 좋은 UX로 이어진다는 것을 확인했다.
